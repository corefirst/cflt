## 1. LLM 作为桥梁：CFLT 的第二支柱

在 CFLT 框架中，LLM 不仅仅是翻译工具；它们是使协议落地的标准化**推理引擎**。CFLT 依靠 LLM 来执行两个不同的任务：

1. **逻辑转换器（Logic Transformer）：** 将凌乱的用户输入转换为严格的 `[核心] → [理由] → [空间] → [时间]` 序列。
2. **语法叠加层（Grammar Overlay）：** 将该严格序列润色为地道的母语级输出（例如，L2 英语、法语或日语）。

CFLT 的计算依据受到以下观察的启发：**LLM 行为对提示词设计敏感**。Habba 等（2025）显示，联合改变多个提示词维度（措辞、分隔符、枚举符、示例、选项顺序）会产生显著的性能波动；Zheng 等（2024）记录到 LLM-as-judge 任务中的顺序效应。需注意，这些工作同时改变*多个*提示词维度，而非单独隔离信息顺序。CFLT **假设**：强制执行固定的、以核心为锚点的序列可以降低模型输出的方差并提高其指令遵循的可靠性；这是一个待检验的预测（见 §10），而非上述敏感性文献已确立的结果。

## 2. Transformer 注意力与位置偏置

Transformer 架构（Vaswani 等，2017）将输入序列视为一组 Token，其中每个 Token 都有可能注意到其他任何 Token。然而，实证研究表明，注意力的分布是**不均匀的**。

### 2.1 首因效应与位置 0
LLM 表现出明显的**首因效应（Primacy Effect）**：处于提示词最开头的信息对模型的内部状态具有不成比例的巨大影响。

### 2.2 位置编码
无论是绝对位置编码（Vaswani 2017）、相对位置编码（Shaw 等，2018）还是旋转位置编码（Su 等，2021），位置编码都确保了模型知道 Token 的位置。最近的基准测试表明，“绝对位置 0”获得了不成比例的关注（“注意力汇点”，Xiao 等，2024）。

### 2.3 位置 0 的注意力：两个不同现象

现代 LLM 中位置 0 被过度关注是出于**两个不同原因**，必须区分：

1. **注意力汇点（Xiao 等 2024）** —— *softmax 稳定性副产物*。因 softmax 分母必须求和为 1，注意力会"泄漏"到那些语义内容**并不**特别相关的早期 token 上。Xiao 等明确指出这些 token *"not being semantically important"* —— 汇点是 softmax + 窗口注意力的数学后果，而**不是**语义优先性的信号。
2. **首因/位置偏置** —— 早期 token 也因每个后续 token 的查询都先看到它们而被多关注（因果掩码在深度上累积）。这与汇点机制独立，**确实**支持把语义丰富的内容放在前面。

**CFLT 实际利用的是什么。** CFLT 的 Core-first 主张依靠的是 **(2) 首因**，而不是 (1) 汇点。因为听者/模型在所有后续处理中都条件于前几个 token，把显著性锚点置于此处会在生成的其余部分中累积影响力。汇点现象是一个独立的工程发现 —— 关于*模型在长上下文中如何保持稳定* —— 它本身**不**主张把语义内容置于位置 0。

还需注意：把 Core 放在位置 0 **并不会**"消耗"汇点 —— 多数现代系统把第一个 token（通常是 `<bos>`）作为专用汇点槽，而 CFLT 的 Core 会占据其紧后位置。关键的论断只是：*高注意力前缀区最好被高信息量内容占用。*

```mermaid
graph BT
    subgraph "Transformer 注意力权重分布"
    P0[位置 0：核心] --- W0((高权重))
    P1[位置 1：理由] --- W1(.)
    P2[位置 2：空间] --- W2(.)
    P3[位置 3：时间] --- W3(.)
    end
    style W0 fill:#f96,stroke:#333,stroke-width:2px
    Note[高注意力前缀区：首因 + 汇点共同作用]
```

（关于互补效应 —— 即*序列中间*的 Token 会被系统性地关注不足 —— 见下文 §3 关于“迷失在中部”现象的讨论。）

### 2.4 锚定非动作核心

> **小节状态。** 下面两条是**与 §2.3（首因）一致、外推到身份核心与请求核心的架构性预测；它们尚未在非动作核心上被直接测量。** §2.3 所引文献（Liu 等 2023；Xiao 等 2024；Lu 等 2022）使用的是检索与分类任务，而非身份/请求分解。机制是同一个首因效应，但其在非动作核心上的*结果量级*仍是开放的经验内容（见 §10.1）。"定义性锚点"是项目内部用词 —— 我们刻意避免"定义性汇点"，因为注意力汇点副产物（Xiao 等 2024）按其本身的分析在语义上是无差别的（§2.3）；为语义内容主张复用"汇点"一词会重新引入 §2.3 所否认的框架。

首因效应启发出以下针对任何置于位置 0 的高显著性成分（不仅限于动词）的预测：

- **身份核心（预测）：** 将“主语身份”（例如，“The solution is...”）放在位置 0，预期会在高注意力前缀区创建一个**定义性锚点**。随后的修饰语（“如何”和“为什么”）将通过这一固定的谓词来解释，我们预测这会降低模型在长段描述中丢失主语身份的倾向。这是一个由首因推出的预测，而非已测得的结果。
- **请求核心（预测）：** 将指令（例如，“Please summarize...”）放在开头，预期会使**任务算子**在早期 KV 缓存中更稳定。我们预测这会降低那种中间信息过重的提示词导致模型开始描述文本而非执行所请求动作的“任务漂移”（项目内部标签，非文献术语）。同样，这是由首因推出的预测，而非已测得的结果。

> **非单调位置偏置警示。** 首因优势是**任务相关且非单调的**，并非普遍偏向首位。Pezeshkpour & Hruschka (2024) 显示，**重排多选题选项（选项内容保持不变）**时，LLM 准确率波动可达数十个百分点。他们的研究针对的是 MCQ 选项顺序，而非话语成分顺序，且并未确立一种通用的、模型层面的首位/末位/中间偏好分类；其发现是：当模型在前几个候选答案之间不确定时，选项位置与候选间距可改变其决策。Zheng et al. (2024) 在 LLM-as-judge 任务上记录到类似的非单调位置效应。**对于自然叙事顺序本身即推理脚手架的任务（如以"备选 → 最终决策"形式呈现的多候选决策），把结论前置可能在部分模型上*移除*脚手架而损害准确率。** 这一反效应在 CFLT 实测中是**模型特异的**而非普适的（详见 `methodology/llm-part2-verification.md` §2.3）：在 L4 多动作决策场景下，DeepSeek V4 Pro 显示 CFLT 比 control 低 11pp，但其他四个被测前沿模型（GPT-5、Gemini 3 Flash、Qwen3.5-Plus、Claude Sonnet 4.6）的 L4 在两组上都接近或处于 100% 天花板——因此 L3-vs-L4 反向观察目前仅限于一个模型。协议层首因主张因此在被测范围内成立，L4 多候选限制现在最适合刻画为 **DeepSeek-specific 模型 × 任务类型交互**，而非 CFLT-on-buried-decisions 的通用属性。更深入的讨论见 §10.4–10.5。

## 3. "迷失在中部"现象 (Lost-in-the-Middle)

Liu 等人（2023，*Lost in the Middle*）证明了，在所测试的长上下文任务上，LLM 的**行为表现**常呈 U 型曲线：对于长上下文开头或结尾附近的信息，准确率往往较高，而对于中间的信息则较弱。这是一种行为表现模式；Liu 等测量的是任务准确率，并未把注意力、因果掩码或位置编码隔离为其机制，且该模式在不同模型、任务和上下文长度上强度不一。

> **重要 —— 原始发现的尺度。** Liu 等的实验使用**多文档问答与键值检索**，把文档/键值放在长上下文中不同位置（10-30 个文档）。他们设置中的"位置"指的是*长上下文中的文档位置*，不是*单句内的 token 位置*。从"文档尺度的中部丢失"直接外推到"句子尺度的核心动作"是*跨尺度类比*，应作为类比对待。

CFLT 在**两个不同尺度**应用 lost-in-the-middle 发现，推论强度不同：

1. **文档/提示词尺度（强支持）。** 当 CFLT 结构化内容置于长 agentic 提示词或 RAG 上下文内时，把最关键的 Core 块放在提示词开头是 Liu 等发现的直接应用。这针对的是**已检索上下文喂给生成器**时的排序；它并不涉及 Core-first 表述是否能改善**检索**步骤本身（top-K 召回）——后者 Lewis et al.（2020）并未测试，需另做检索器特定的 CFLT/对照基准。
2. **句子/token 尺度（类比性，现已初步实测）。** 单句内，相关现象是*首因*+*注意力汇点*（参见 §2.3），不是 lost-in-the-middle。U 型曲线在句内 token 粒度上尚未确证。**Part II 抽取 pilot**（[`../methodology/llm-part2-verification.md`](../methodology/llm-part2-verification.md)）现提供初步的句级证据：5 个前沿模型上，Core-first 提示词把重干扰（L3）抽取准确率较自然 reason-first 语序提高 **+22–44pp**。这**强化、但未坐实**句子级主张（caveat：小 N；单一对照序；仅抽取任务）。把句子级主张视为由文档级发现*启发*、并由 Part II pilot **初步支持**。

通过将**核心动作**放在最开头，CFLT 确保信息中最关键的部分占据高注意力前缀区。修饰语（理由、空间、时间）占据随后的槽位，对于典型的句子长度输入，这些槽位仍然处于高注意力的早期窗口内。

```mermaid
graph LR
    subgraph "U 型信息保留曲线"
    Start[提示词开头] -- "高准确率" --> High1((.))
    Middle[提示词中部] -- "低准确率" --> Low((.))
    End[提示词末尾] -- "高准确率" --> High2((.))
    end
    
    Start --- Middle --- End
    
    Note[注：CFLT 将“核心”放在开头以确保最大的关注度]
```

## 4. 提示词引导与自回归预测

LLM 是自回归的：它们根据所有前面的 Token $(t_1, \dots, t_{n-1})$ 来预测下一个 Token $t_n$。

$$
p(t_n \mid t_{n-1}, \dots, t_1)
$$

如果 $t_1, t_2, \dots$（前缀）是高熵、低相关性的 Token（如长难句“昨天当我正在走路的时候……”），模型对于核心动作的状态约束就很弱。如果前缀是**核心动作**本身，那么后续所有槽位的概率分布会立即收窄。

CFLT 充当了一种**引导协议（steering protocol）**，在生成过程的早期使模型的分支因子坍缩。我们**预测**这有助于产生更受约束的延续并减少幻觉漂移；其量级是 §10.1 的开放经验问题，而 §9 *局限性说明* 第 5 条讨论了在哪些反对立场（Min 2022 / Sclar 2024 / 端到端基线）下这一预测可能小于预期。

## 5. 语境学习与“CFLT 流形”

语境学习（In-Context Learning, ICL）通过向模型提供可以扩展的模式来工作（Min 等，2022）。

- 传统的语法规则很难通过少量示例（few-shots）来表达。
- **CFLT 协议**是一个简单的线性模式。

CFLT **假设**：由于 `[核心] → [修饰语]` 序列接近模型训练数据中已有的模式，模型可以通过极少数示例（低样本，或仅通过简单的系统提示词进行零样本）学会遵循该协议。没有任何被引来源直接测试 CFLT 序列的可学习性；低样本/零样本下的 CFLT 遵从是一个开放的经验问题，需在零样本、少样本和微调条件下分别测量（见 §10）。

## 6. Token 经济与计算成本

最近关于结构化提示词（TOON、CSV 等）的研究报告称，与冗长的自然语言或密集的 JSON 相比，将信息展平为线性的、非嵌套的格式可以显著减少 Token 消耗。报告的幅度因领域而异 —— 已发布的结构化数据基准测试显示，表格内容的降幅在 30%–50% 之间。**CFLT 特有的降幅尚未测量**，不应直接假定与结构化数据文献相匹配：CFLT 处理的是话语语义，而非表格字段，其降幅源自不同的源头（消除句法协调开销，而非序列化行数据）。

CFLT 可能通过以下方式贡献于 Token 经济：
1. **线性化：** 消除对复杂句法标记（关系代数、嵌套从句）的需求。
2. **显式空值（Nulls）：** 对缺失的槽位使用“NULL” Token，防止模型为了保持语法流利而生成“填充”文本。
3. **前缀缓存（Prefix Caching）：** 推理引擎（vLLM, SGLang）仅对**逐 token 完全相同的静态前缀**复用 KV 状态。因此可复用部分是固定的 `[系统提示词] + [schema]` 脚手架；`[核心]` 本身通常随请求而变，无法跨不同请求缓存。任何计算节省都取决于共享前缀长度、负载和引擎配置，必须实测，而不能直接继承引擎基准报告的数字（见 `methodology/llm-prompting.md`）。

```mermaid
graph TD
    subgraph "请求 A"
    S1[系统 + Schema] --> C1[核心 A] --> M1[修饰语 X]
    end
    
    subgraph "请求 B"
    S2[系统 + Schema] --> C2[核心 B] --> M2[修饰语 Y]
    end
    
    Cache[KV 缓存：仅系统 + Schema]
    S1 --- Cache
    S2 --- Cache
    
    Cache -- "已复用：逐 token 相同的静态前缀" --> C1
    Cache -- "已复用：逐 token 相同的静态前缀" --> C2
```

实际的降幅必须通过 [`methodology/evaluation-metrics.md`](../methodology/evaluation-metrics.md) §4.1 中指定的消融实验来量化。

## 7. 幻觉动态 (Hallucination Dynamics)

CFLT **假设**一种特定的失败模式：某些幻觉发生在模型丢失了主要断言（核心）并开始生成听起来合理但无关的上下文时。这一"幻觉即核心丢失"的说法是 CFLT 提出、有待检验的框架；被引的摘要忠实性文献（如 Maynez 等 2020）记录了系统会生成不忠实内容，但**并未**把成分顺序、或核心丢失确立为其原因。

通过将**核心**置于位置 0，CFLT 利用首因效应（参见 §2.3）在早期 KV 缓存中提供稳定的"语义锚点"。**我们预期**这会降低模型随着序列增长而偏离用户意图的"漂移"风险；降幅量级是一个开放的经验问题（见 §10.1 *具体准确度增量*）。（注：这是首因论证；§2.3 讨论的注意力汇点副产物是*独立*现象，并非本主张的依据。）

## 8. 潜空间中的跨语言对齐

由于 LLM 是在海量的多语言语料库上训练的，CFLT **假设**它们为语义概念跨语言地开发了一个相当程度**共享的潜空间（latent space）**。完全*语言中性*表征的存在并未被多语言评估文献确立——该文献记录的是不均衡的跨语言行为，而非表征上的中性；因此我们把语言中性视为假设，而非事实。

CFLT 通过使用**与语言无关的序列**来瞄准这个潜空间。其设计意图是：无论表面 Token 是中文、英文还是阿拉伯文，进入注意力头的*概念顺序*都相同。关于在被测范围内采用单一成分顺序的类型学动机——四个语系的五种语言，每种 Core 内部组装机制都有参考语法引用——见 [`core-concept.md`](./core-concept.md) §2.5。这是作为项目标准的设计选择提出的，而非已证实的跨语言不变性。CFLT 提议把 LLM 作为实现人类学习者用来连接不同语言的“中性缓冲区”的工具。

## 9. 局限性说明

1. **严格性 vs. 流利度：** 强制执行严格的序列有时会导致较小模型的输出显得“生硬”。语法叠加层对于恢复自然流利度至关重要。
2. **指令遵循：** 极小的模型（如参数量 <3B）在没有微调的情况下，可能难以维持严格的 CFLT 协议。
3. **推理 vs. 线性化：** 虽然 CFLT 改善了话语结构，但它并不能替代用于复杂数学或逻辑问题的思维链（CoT）推理（Wei 等，2022）。它应该与 CoT *结合使用* —— CFLT 为那些推理链本身不是核心（只有最终行动/决定是核心）的任务提供了稳定的位置 0 锚定；CoT 则提供了 CFLT 未指定的中间步骤结构。
4. **长上下文漂移：** 即使有首因前缀锚定，极长的修饰语仍可能导致模型丢失核心。建议进行模块化处理（将想法拆分为多个 CFLT 句子）。
5. **如果内容顺序的重要性不如我们声称的那么大怎么办？** 三个严肃的反对立场威胁着 LLM 端的论点；我们在这里直接面对它们：
    - **Min 等 (2022) "Rethinking the Role of Demonstrations"** 发现，在语境学习中，示例中*输入-标签映射的正确性*惊人地不重要 —— 示例之所以有用，主要在于它指定了*示例格式*、*标签空间*和*输入分布*。（Min 等研究的是示例*属性*；他们**并未**确立示例顺序效应——那一结果属于相邻工作，例如 Lu 等 2022 关于少样本示例顺序。）对于任何简单的“位置 0 内容至关重要”的主张，这是一个部分反证。CFLT 的回应是，CFLT 作用于*格式/架构*轴（Min 等显示该维度有效），而不是主要作用于任一 token 的事实正确性 —— 该协议的规定是“固定架构和说话者承诺的线性顺序”，而不是“确保核心 token 是提示词中最准确的事实”。`core-concept.md` §4.1 / §8.5 中对 Min 的引用应根据此解读。
    - **Sclar 等 (2024) "Quantifying Language Models' Sensitivity to Spurious Features in Prompt Design"** 发现，LM 的准确率随分隔符、格式和*非语义*特征发生剧烈波动。这是一个威胁：如果虚假的表面格式化可以主导内容顺序，那么 CFLT 内容顺序纪律的边际收益就不明确。CFLT 的回应是，固定*架构（格式化）*和*内容顺序*才是其贡献所在 —— 这两个轴是互补的，CFLT 最能被辩护为一种格式与内容的联合纪律，而非单纯的内容顺序主张。
    - **端到端指令微调基准。** 随着前沿指令微调模型 (GPT-5.5, Claude 4.7-class, Gemini 3.1+) 的规模扩大，显式线性化支架的边际收益可能会缩小 —— 对于许多任务，直接对前沿模型使用一行“按 [核心, 理由, 空间, 时间] 的形式回答”指令，可能与单独的 CFLT 预处理步骤相当。CFLT 的回应是 **P2 的证伪条款** (`foundations/core-concept.md` §8.5)：若在 ≥ 3 个前沿模型家族中，CFLT 格式的提示词在位置 0 未显示可测量的注意力或准确率优势，则 LLM 端的主张被推翻。我们接受这一证伪条件。
6. **预测与测量状态。** 本文档中的几个机制主张 (例如 §2.4 身份/请求核心预测，§7 幻觉漂移减少) 是基于 §2.3 首因框架的外推，而非测得的 CFLT 特有结果。衡量这些结果的实验议程见 [`../methodology/empirical-agenda.md`](../methodology/empirical-agenda.md)。

## 10. 开放的研究问题

1. **具体准确度增量：** 在“大海捞针（Needle-in-a-Haystack）”基准测试中，CFLT 与自由格式提示词相比，在指令遵循方面的绝对提升是多少？
2. **延迟影响：** 在生产级 RAG 系统中，通过前缀缓存 CFLT 结构可以节省多少 TTFT（首个 Token 时间）？
3. **微调收益：** 在专门针对 CFLT 线性化语料库进行微调的模型上，其在跨语言任务中的表现是否优于标准的指令微调？
4. **非单调位置偏置。** Pezeshkpour & Hruschka (2024) 显示 LLM 在多选题上的准确率对*选项顺序*（选项内容保持不变）的敏感性是非单调的——首位并非一致占优。他们报告：选项位置与候选间距尤其在模型不确定时改变其决策；他们并未确立通用的、模型层面的首位/末位/中间分类，且 MCQ 到成分顺序的迁移是间接的。Zheng et al. (2024) 在 LLM-as-judge 评测上记录到类似的非单调位置效应。**这是对 CFLT 首因偏好 Core-first 主张的部分反证**：如果位置 0 优势是任务相关且非单调的，CFLT 严格的 Slot-0 放置可能并非对所有任务最优。CFLT 实测（`methodology/llm-part2-verification.md` §2.3）显示该反效应是**模型特异**而非普适的：L4 多动作场景下 DeepSeek V4 Pro 显示 CFLT 比 control 低 11pp，但其他四个被测前沿模型（GPT-5、Gemini 3 Flash、Qwen3.5-Plus、Claude Sonnet 4.6）的 L4 在两组上都接近或处于 100% 天花板——因此 L3-vs-L4 反向观察目前仅限于单个模型。**开放问题**：DeepSeek L4 回归是 (a) 一种 DeepSeek-V4-Pro 特异的、在多候选 items 上的指令遵从异常（模型 × items 交互；目前与 5-model 数据最一致的读法），(b) 任务类型的属性但被其他模型的天花板效应掩盖（需用更难的 L4 items 让其余四个模型变成 informative），还是 (c) 与 Pezeshkpour & Hruschka 意义下的模型内禀位置先验相关？需要在 ≥ 6 个模型家族 × ≥ 4 种任务类型（抽取、MCQ、摘要、多步骤规划）上做预注册的因子设计，且 item 难度需校准至当前四模型天花板之上，方能区分 (a)–(c)。
5. **逆转诅咒与内禀顺序敏感性。** Berglund et al. (2024) "The Reversal Curse" 显示在 "A is B" 上训练的 LLM 不会自动学到 "B is A"——顺序敏感性与所学表示结构性纠缠，不仅仅是 prompt 时格式的问题。这意味着 **CFLT-as-alignment-constraint**（`../methodology/empirical-agenda.md` §2.2；上面 §10.3 "微调收益"）可能比 CFLT-as-prompt-protocol 面临更难的问题：在 CFLT-重排数据上训练可能不会转化为推理时的 CFLT 鲁棒性，如果模型把 "Core ... Reason ..." 视为与 "Reason ... Core ..." 不同的 token 序列。一个对照微调实验比较 (i) 标准指令微调基线，(ii) CFLT-重排微调，(iii) 双向 CFLT-and-reverse 微调，在 prompt 顺序扰动鲁棒性上评估，可以解决这一问题。我们接受 Berglund et al. 为部分反证：它不反驳 prompt 时 CFLT 主张，但它限缩了 alignment-constraint 升级的范围。
