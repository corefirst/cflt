# CFLT 实验验证计划：LLM 准确性与效率

> **版本：** 1.0.0  
> **目标：** 在大语言模型（LLM）上验证核心优先语言理论（CFLT）。  
> **状态：** 定稿。

---

## 1. 目标与范围
本计划详细说明了一个严密的实证框架，用于验证大语言模型（LLM）背景下的**核心优先语言理论（CFLT）**协议。目标是证明将说话者的主要承诺（“核心”）锚定在位置 0（Position 0）可以显著提高模型性能，同时优化计算成本。

---

## 2. 验证矩阵（分级难度）
为了确保稳健性，验证套件分为五个信息密度和逻辑复杂度逐渐增加的级别。每个测试用例都将**母语表达（对照组）**与**CFLT 结构化表达（实验组）**进行对比。

### 第 1 级：简单意图（低密度）
*   **描述：** 单一动作，单一修饰语。
*   **对照组（母语）：** “请关上窗户，因为外面在下雨。”
*   **CFLT：** “[CORE] 关窗, [REASON] 因为下雨。”
*   **预期 JSON：** `{"action": "close", "target": "window"}`

### 第 2 级：多参数（中等密度）
*   **描述：** 具有特定空间、时间和原因槽位的单一动作。
*   **对照组（母语）：** “既然会议已经结束，请在今天下午 6:00 关闭 5 楼大会议室的所有灯。”
*   **CFLT：** “[CORE] 关灯, [REASON] 会议结束, [SPACE] 5F 大会议室, [TIME] 今天 18:00。”
*   **预期 JSON：** `{"action": "off", "target": "lights", "location": "5F_conference_room", "time": "18:00"}`

### 第 3 级：高密度 / 嘈杂上下文（高难度）
*   **描述：** 核心动作埋藏在一段背景信息和无关细节中（测试“中间迷失”现象）。
*   **对照组（母语）：** 一段描述服务器房维护和温度波动的叙述，最后请求关闭备份节点。
*   **CFLT：** “[CORE] 关闭备份节点, [REASON] 防止硬件损坏, [SPACE] 东翼服务器房, [TIME] 14:15。”
*   **预期 JSON：** `{"action": "shutdown", "target": "backup_nodes", "priority": "high"}`

### 第 4 级：复杂显著性 / 意图提取（极高难度）
*   **描述：** 提到多个潜在动作；模型必须识别出*主要的*承诺。
*   **对照组（母语）：** “我想过报警，但我决定今晚留在家里整理证据，而不是明天去保险公司。”
*   **CFLT：** “[CORE] 整理证据, [REASON] 避免冲突升级, [SPACE] 家里, [TIME] 今晚。”
*   **预期 JSON：** `{"primary_action": "document_evidence", "status": "decided"}`

### 第 5 级：高密度数据与多步逻辑（计算/极限难度）
*   **描述：** 原始未格式化的日志提取 + 多步算术 + 条件逻辑约束。
*   **对照组（母语）：** 一段原始服务器日志片段，随后是一个复杂的请求，要求计算平均 CPU 使用率并过滤符合特定 CPU/错误/内存阈值的条目。
*   **CFLT：** “[CORE] 计算平均 CPU 并提取高负载错误日志, [REASON] 防止内存溢出 (限制:30GB), [SPACE] 日志数据, [TIME] 过去一小时。”
*   **预期 JSON：** 包含 `avg_cpu`、`high_load_errors` 和布尔值 `melting_point_triggered` 的汇总指标。

## 3. 评估模型与硬件标准

### A 类：本地开源模型（24-32G RAM 可及性）
*目标：确保社区可以在标准消费级硬件（如 RTX 3090/4090 或 32GB Mac）上复现实验。*
*   **Meta-Llama-3-8B-Instruct**（全球生态基准）
*   **Qwen2-7B-Instruct**（领先的中英/逻辑开源模型）
*   **DeepSeek-V2-Lite-Chat / DeepSeek-Coder-V2-Lite**（高效的 MoE 架构）
*   **Gemma-2-9B-It**（Google 最新的先进开放权重模型）
*   *量化：4-bit (AWQ/GGUF) 或 8-bit (FP8) 以适应 16GB-24GB 显存。*

### B 类：前沿闭源模型（2025-2026 年度最高水平）
*   **OpenAI:** `gpt-5.5-pro`, `gpt-5.4-mini`
*   **Anthropic:** `claude-opus-4-7`, `claude-sonnet-4-6`
*   **Google:** `gemini-3.1-pro-preview`, `gemini-3-flash`

## 4. 验证指标（数据驱动对比）

评估基于 **API 响应元数据**和**生成内容**的直接对比。

### 指标 1：准确率（JSON 命中率）
*   **机制：** 将模型输出直接与**预期 JSON 标准答案**进行对比。
*   **成功标准：** 键和值的精确匹配（二进制 1 或 0）。
*   **目标：** 证明 CFLT 具有更高的命中率，尤其是在 Level 3–5 的情况下。

### 指标 2：输入效率（Prompt Token 消耗）
*   **机制：** 从 API 响应中提取 `usage.prompt_tokens`。
*   **目标：** 证明 CFLT 协议减少了“语法冗余”，从而降低了单次请求成本。

### 指标 3：输出质量（Completion Token 浪费）
*   **机制：** 提取 `usage.completion_tokens`。
*   **目标：** 证明 CFLT 的结构清晰度防止了“模型啰嗦”，从而产生了简洁、直击要点的 JSON 响应。

---

## 5. 参数设置（通用）
*   **Temperature:** `0.0`（确定性输出）
*   **Response Format:** `json_object`（强制要求）
*   **Max Tokens:** `512`
*   **Top_P:** `1.0`

---

## 6. 执行流程
1.  **数据集准备：** 为 5 个难度级别中的每一个生成 50+ 对样本。
2.  **顺序执行：** 分别通过目标模型运行提示词（无并发压力测试）。
3.  **对比分析：** 生成一份报告，对比所有难度级别和模型类别的命中率及 Token 使用情况。
